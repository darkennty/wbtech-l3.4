# Image Processor

Микросервисное приложение на Go для загрузки, асинхронной обработки и хранения изображений. Проект использует Apache Kafka в качестве брокера сообщений для разделения HTTP-сервера и воркера, выполняющего тяжелые задачи (ресайз и наложение водяных знаков).

## Функционал

- **Асинхронная архитектура:** Загрузка файла и его обработка разделены. Пользователь мгновенно получает ID картинки, пока Worker трудится в фоне.
- **Обработка фото:**
  - Изменение размера основного фото (до 1024px).
  - Наложение кастомного текстового водяного знака.
  - Генерация миниатюры (Thumbnail 200x200px).
- **Поддержка форматов:** Строгая валидация на стороне сервера (допускаются только `.jpg`, `.jpeg` и `.png`).
- **Web-интерфейс:** Встроенный современный фронтенд (HTML/JS/CSS) для загрузки картинок, отслеживания статуса в реальном времени (Polling) и просмотра результатов.
- **Умное хранилище:** При удалении картинки по ID сервер автоматически зачищает БД и все 3 физических файла с диска (оригинал, обработанную версию и миниатюру).
- **Кликабельные превью:** На странице отображаются три превью (оригинал, с водяным знаком, миниатюра), которые при клике открываются в полном размере в новой вкладке.
- **Сохранение сессии:** Список загруженных ID хранится в `localStorage`, поэтому при перезагрузке страницы вы не потеряете свою работу.

## Архитектура

Приложение разделено на два основных сервиса, которые общаются через брокер сообщений Kafka, что обеспечивает отказоустойчивость и масштабируемость.

1.  **API Server (api)**: Веб-сервер на Gin, который принимает HTTP-запросы.
    - Принимает файл, сохраняет его оригинал.
    - Создает запись в PostgreSQL со статусом `pending`.
    - Отправляет задачу на обработку в топик Kafka.
    - Отдает статус обработки и готовые изображения по ID.
2.  **Worker (consumer)**: Фоновый обработчик (Kafka Consumer).
    - Подписывается на топик Kafka и получает задачи.
    - Выполняет ресурсоемкие операции: ресайз, наложение водяного знака, создание миниатюры.
    - Сохраняет обработанные файлы.
    - Обновляет статус задачи в PostgreSQL на `completed` или `failed`.
    
## Сборка и запуск

1.  Клонируйте репозиторий:
    ```bash
    git clone https://github.com/darkennty/wbtech-l3.4.git
    cd wbtech-l3.4
    ```

2.  Создайте файл `.env` из примера и заполните его своими данными:
    ```bash
    cp .env.example .env
    ```

3.  Запустите все сервисы с помощью Docker Compose:
    ```bash
    docker-compose up --build
    ```

4. Запустите программу:
    ```bash
   go run cmd/app/main.go
   ```

5. Откройте браузер и перейдите по адресу `http://localhost:8080`.

## API Эндпоинты

| Метод  | Путь             | Описание                                                                                                |
| ------ | ---------------- | ------------------------------------------------------------------------------------------------------- |
| `POST` | `/upload`        | Загрузка изображения. В `form-data` должен быть ключ `image`.                                           |
| `GET`  | `/image/:id`     | Получение статуса или файла изображения. Принимает query-параметр `type` (`original`, `processed`, `thumb`). |
| `DELETE`| `/image/:id`    | Удаление изображения (запись в БД и все 3 файла).                                                         |

## Переменные окружения

Все необходимые переменные окружения находятся в файле `.env.example`.

| Переменная          | Описание                                    |
| ------------------- | ------------------------------------------- |
| `API_PORT`          | Порт, на котором работает API-сервер        |
| `POSTGRES_USER`     | Имя пользователя для PostgreSQL             |
| `POSTGRES_PASSWORD` | Пароль для PostgreSQL                       |
| `POSTGRES_DB`       | Название базы данных                        |
| `POSTGRES_HOST`     | Хост PostgreSQL (внутри Docker сети)        |
| `POSTGRES_PORT`     | Порт PostgreSQL                             |
| `KAFKA_BROKERS`     | Адреса брокеров Kafka через запятую         |
| `KAFKA_TOPIC`       | Название топика для задач обработки         |
| `KAFKA_GROUP_ID`    | ID группы консьюмеров                       |
| `WATERMARK_TEXT`    | Текст для водяного знака                    |

## Структура проекта

Проект имеет модульную структуру, основанную на принципах Clean Architecture.

```
.
├── cmd/                # Точки входа для сервисов (api и consumer)
├── config/             # Конфигурация приложения (Viper)
├── internal/           # Внутренняя логика, недоступная извне
│   ├── handler/        # HTTP-хендлеры (Gin)
│   ├── kafka/          # Логика для Producer и Consumer
│   ├── model/          # Модели данных (сущности)
│   ├── processor/      # Ядро обработки изображений
│   ├── repository/     # Работа с базой данных (PostgreSQL)
│   ├── service/        # Бизнес-логика
│   └── storage/        # Работа с файловым хранилищем
├── migrations/         # Миграции базы данных (goose)
├── pkg/                # Переиспользуемые пакеты (например, logger)
├── static/             # Фронтенд-файлы (HTML, CSS, JS)
├── .env.example        # Пример файла с переменными окружения
├── docker-compose.yml  # Файл для оркестрации контейнеров
└── Dockerfile          # Docker-файлы для сборки образов
```
